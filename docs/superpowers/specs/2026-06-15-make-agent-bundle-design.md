# Design: make-agent-bundle (Priority 2)

Date: 2026-06-15
Status: Approved
Author: Ming (Claude Code)

## Context

`export_synthetic()` already produces a complete bundle (synthetic_data.csv,
data_dictionary.csv, ai-readme.md, privacy_report.txt, manifest.json,
load_data.R). The gap is:

1. No single-command orchestration from raw file to bundle
2. No `diagnostic_view.json` — the machine-readable index agents use to
   understand what they received without unzipping

Primary caller: AI agents via CLI. Human analysts are secondary.

## Architecture

New file `R/make-agent-bundle.R` exports one function:

```r
make_agent_bundle(file, out, purpose = "ai_programming", seed = NULL,
                  overwrite = FALSE, ...)
```

Fixed pipeline (no user-configurable steps):

```
read_input(file, ...)
  → profile_data()
  → detect_roles()
  → synth_spec(purpose = purpose, seed = seed)
  → synthesize_data()
  → export_synthetic(..., include_report = FALSE, format = "dir",
                     path = temp_bundle_dir)
  → write diagnostic_view.json into temp_bundle_dir
  → zip temp_bundle_dir → out
  → clean up temp_bundle_dir
```

`make_agent_bundle()` calls `export_synthetic(format = "dir")` into a
temp directory it controls, then injects `diagnostic_view.json` before
zipping manually with `zip::zip()`. This avoids modifying `export_synthetic()`.

`treatment` values for `diagnostic_view.json` are read from the
`data_dictionary.csv` that `export_synthetic()` writes into the temp dir —
no duplication of the treatment inference logic.

`...` passes through to `read_input()` only (encoding, sheet, etc.).
All synthesis decisions come from the purpose preset.
`include_report = FALSE` — agents don't need the HTML report, and it
requires optional packages.

### CLI command

Thin wrapper added to `cli.R`:

```
dataganger make-agent-bundle <data-file> --out <bundle.zip>
                             [--purpose <purpose>] [--seed <n>]
```

- `--out` is required; aborts with exit 2 if missing
- `--purpose` defaults to `ai_programming`
- `--seed` optional integer
- Exit 0 success, 1 processing error, 2 bad arguments
- Consistent with existing CLI conventions

## diagnostic_view.json

Generated from roles tibble + spec + synthetic dims. Written into the
bundle directory before `export_synthetic()` zips everything.

```json
{
  "source": "dataganger",
  "dataganger_version": "0.2.0",
  "purpose": "ai_programming",
  "dataset": {
    "n_rows_bucket": "100-999",
    "n_cols": 8
  },
  "columns": [
    {
      "name": "patient_id",
      "role": "ID candidate",
      "sensitive": true,
      "treatment": "synthesized"
    },
    {
      "name": "notes",
      "role": "free text",
      "sensitive": true,
      "treatment": "free_text_dropped"
    }
  ],
  "blocked": {
    "raw_rows": true,
    "free_text_examples": true,
    "ids_synthesized": true,
    "plots": true
  }
}
```

### n_rows_bucket bands

| Actual rows | Bucket label |
|-------------|--------------|
| < 100       | `<100`       |
| 100–999     | `100-999`    |
| 1000–9999   | `1000-9999`  |
| 10000–49999 | `10000-49999`|
| ≥ 50000     | `50000+`     |

### treatment values

Reuses values already present in data_dictionary.csv:
`synthesized`, `free_text_dropped`, `masked_or_dropped`, `renamed`,
`schema_only`, `coarsened_date`.

### blocked fields

Derived from roles and spec, not user-supplied:

- `raw_rows`: always `true`
- `free_text_examples`: `true` when any column has role `free text`
- `ids_synthesized`: `true` when any column has role `ID candidate`
- `plots`: always `true`

## Error handling

Fails early and loudly — no silent fallbacks, no partial output:

- `file` not found → `cli_abort` (from `read_input`)
- `out` parent directory missing → `cli_abort` before any work starts
- `out` exists and `overwrite = FALSE` → `cli_abort` (from `export_synthetic`)
- synthesis produces 0 rows → `cli_abort`
- any step throws → propagates as-is; CLI catches and exits 1
